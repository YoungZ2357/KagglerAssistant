# 开发备忘




## 工具开发备忘

### 向工具注入状态
> 使用情景：工具需要获取状态值而不仅仅是传参
> 
使用 `state: Annotated[dict, InjectedState]`，以使得LangGraph引擎自动填充状态。Agent不会对这个参数作出任何操作。
期望LangGraph自动使用时，必须使工具被绑定在ToolNode，否则需要手动传入状态值(如期望脱离图运行或者于自定义的节点运行时)

#### 正确示例
```python
@tool
def explore_schema(state: Annotated[dict, InjectedState]) -> str:
    ...
```

### 关于工具docstring
>使用情景：任何顶层工具的函数级标注

工具函数(带有`@tool`修饰符)的docstring是LLM感知工具功能的重要途径之一，不要使用标准的Google风格docstring，直接编写工具描述和应用场景


#### 正确示例
```python
@tool
def explore_schema(state: Annotated[dict, InjectedState]) -> str:
    """获取数据集的完整结构信息：列名称、数据类型、缺失数量、唯一值数量和示例值。
    使用情景：
    - 用户要求获取列名、数据类型或者数据集结构
    - 需要进行统计分析但不确定具体列名或数据类型时，先调用此工具确认
    不要使用此工具进行实际的统计分析或分布计算。
    """
    ...
```
### 让工具操作状态
> 使用情景：工具需要更新状态

该条只对产生副作用的工具有效。让顶层工具返回`langgraph.types.Command`，然后编写状态更新即可。总而言之，我们需要这些东西才能让工具来
干涉状态：
- 参数带有`Annotated[dict, InjectedState]`以注入状态（通常需要得到状态是什么
- 参数带有工具调用id，以追溯工具和应用状态更改

#### 正确示例
```python

@tool
def execute_empty_value(
    state: Annotated[dict, InjectedState],
    tool_call_id: Annotated[str, InjectedToolCallId],
    pairs: list[dict],
) -> Command:
    """对指定列执行空值填充或删除操作。

    pairs 是一个列表，每个元素为 {"column": <列名>, "action": <填充方法>}。
    支持的 action 值：
    - "zero"：用零值填充（数值填 0，字符串填 "0"，布尔填 False）
    - "avg"：用均值填充（仅限数值列）
    - "median"：用中位数填充（仅限数值列）
    - "mode"：用众数填充
    - "delete"：删除包含空值的行

    使用情景：
    - 用户指定某些列存在空值并要求处理时
    - 用户可以混合使用多种填充方法，例如某列用均值、另一列删除
    - 当你拥有足够自主权，且认为需要堆数据进行相关处理
    """
    df = data.get(state["data_version"])
    result = exec_empty(df, pairs)

    if "error" in result:
        return Command(update={
            "messages": [
                ToolMessage(
                    json.dumps(result, ensure_ascii=False),
                    tool_call_id=tool_call_id,
                )
            ],
        })

    new_version = data.add_version(result["processed_df"])
    return Command(update={  # 直接返回update
        "data_version": new_version,
        "messages": [
            ToolMessage(
                json.dumps({
                    "new_data_version": new_version,
                    "rows_before": result["rows_before"],
                    "rows_after": result["rows_after"],
                    "preview": result["preview"],
                    "summary": result["summary"],
                }, ensure_ascii=False),
                tool_call_id=tool_call_id,
            ),
        ],
    })
```
## 新增工具开发流程

### 工具开发

新增一个会产生新数据版本的工具，需要修改以下 **6 个文件**，以 `xxx` 为工具/kind 名称示例：

---

#### 1. `src/kaggler/ir/schema.py` — 注册 IR kind

在 `KINDS` frozenset 中添加新的 kind 字符串：

```python
KINDS = frozenset({
    ...
    "xxx",   # 新增
})
```

这是所有 kind 的单一事实来源，解释器和代码生成器都依赖此集合校验合法性。

---

#### 2. `src/kaggler/modes/feature_engineering/compute.py` — 实现拟合 + IR 组装

新增 `exec_xxx()` 函数，负责：

1. 验证输入参数
2. 对 DataFrame 执行拟合（计算统计量如均值、标准差、标签映射等）
3. 将拟合结果冻结为 `IRSpec`
4. 通过 `op_from(ir.kind, ir.params)` 从 IR 构建 Op（复用解释器，保证运行时与恢复路径一致）
5. 应用 Op 得到结果 DataFrame
6. 返回标准 dict

```python
def exec_xxx(df: pl.DataFrame, ...) -> dict:
    # 1. 验证
    # 2. 拟合：计算 frozen_statistics
    # 3. 冻结为 IR
    ir = IRSpec("xxx", {"param1": frozen_statistics, ...})
    # 4. 构建 Op（与 interpret.py 走同一路径）
    _op = op_from(ir.kind, ir.params)
    # 5. 应用
    result_df = _op(df.lazy()).collect()
    # 6. 返回
    return {
        "op": _op,
        "ir": ir,
        "preview": result_df.head(5).to_dicts(),
        "summary": f"处理了 {len(result_df)} 行",
        "rows_before": df.height,
        "rows_after": result_df.height,
    }
```

---

#### 3. `src/kaggler/ir/interpret.py` — 实现 Op 解释器

新增解释器函数并注册到 `INTERPRETERS` dict。解释器从 `IRNode.params` 读取冻结参数，构建 `pl.LazyFrame → pl.LazyFrame` 的闭包：

```python
def _interp_xxx(params: dict) -> Op:
    def _op(lf: pl.LazyFrame) -> pl.LazyFrame:
        return lf.some_operation(params["param1"])
    return _op

INTERPRETERS = {
    ...
    "xxx": _interp_xxx,   # 注册
}
```

**关键不变量**：运行时执行和对话恢复都通过 `op_from()` → `INTERPRETERS` 走同一条路径，保证确定性回放。

---

#### 4. `src/kaggler/ir/emit.py` — 实现代码生成器

新增 emitter 函数并注册到 `EMITTERS` dict。emitter 从相同的 `params` 生成独立可运行的 Polars 代码：

```python
def _emit_xxx(params: dict) -> str:
    return f"lf = lf.some_operation({params['param1']!r})"

EMITTERS = {
    ...
    "xxx": _emit_xxx,   # 注册
}
```

生成的代码会被 `DataProvider.generate_pipeline_code()` 拼接成完整脚本用于导出。

---

#### 5. `src/kaggler/modes/feature_engineering/tools.py` — 暴露 LangChain 工具

新增带 `@tool` 装饰器的函数并在 `make_tools()` 中注册：

```python
@tool
def xxx_tool(
    state: Annotated[dict, InjectedState],
    tool_call_id: Annotated[str, InjectedToolCallId],
    param1: list[str],
) -> Command:
    """工具的 docstring 直接面向 LLM，描述功能、参数和使用情景。"""
    df = data.get(state["data_version"])
    result = exec_xxx(df, param1)
    return commit_mutation(
        data, result, tool_call_id,
        parent_version=state["data_version"],
        tool_name="xxx",
        description=f"执行 xxx: {param1}",
    )

def make_tools(data):
    return [
        ...
        xxx_tool,   # 加入列表
    ]
```

`commit_mutation()` 自动处理：错误返回 ToolMessage、成功时调用 `data.add_version(op, ir=ir)` 将 IR 存入版本树并更新状态。

---

#### 6. 测试 — 验证双投影一致性

在 `tests/test_ir_interpret_emit.py` 中添加测试用例，确保 Op 执行结果与代码生成结果逐帧相等：

```python
def test_xxx_op_runs_and_code_matches(self, df):
    node = _node("xxx", {"param1": ...})
    out = _assert_op_code_equal(node, df)
    assert ...  # 值断言
```

测试模式通过 `_assert_op_code_equal` 自动验证：
- `build_op(node)(df.lazy()).collect()` — Op 闭包执行
- `exec(emit_code(node))` — 生成代码执行
- 两者结果必须 `assert_frame_equal(check_exact=True)`

### 工具注册

底层函数开发完成后，工具需通过链式注册暴露给 LLM。全程 4 层，全部为**声明式注册**（无 XML/YAML 配置，纯 Python 代码）。

---

#### 第 1 层: `make_tools(data)` — 工具工厂（`tools.py`）

通过闭包将 `DataProvider` 注入到 `@tool` 函数中。`make_tools` 接收一个 `data` 参数，返回 `list[BaseTool]`：

```python
def make_tools(data: DataProvider) -> list[BaseTool]:
    @tool
    def execute_empty_value(state, tool_call_id, ...) -> Command:
        df = data.get(state["data_version"])   # data 来自闭包
        result = exec_empty(df, ...)
        return commit_mutation(data, result, ...)

    @tool
    def drop_columns(state, tool_call_id, ...) -> Command:
        df = data.get(state["data_version"])
        result = exec_drop_columns(df, ...)
        return commit_mutation(data, result, ...)

    # ... 其他工具 ...

    return [execute_empty_value, drop_columns, ...]
```

| 关注点                 | 说明                                                              |
|------------------------|-------------------------------------------------------------------|
| 闭包捕获               | `data` 由外层参数传入，内层 `@tool` 函数通过闭包引用               |
| 返回值类型             | `list[BaseTool]` — LangChain 工具对象列表                          |
| 变异工具 vs 只读工具   | 变异工具返回 `Command`（更新 `data_version`），只读工具返回 `str`  |

---

#### 第 2 层: `__init__.py` — 模块导出

```python
from .tools import make_tools
from .prompts import FEAT_ENG_SYSTEM_PROMPT_TEMPLATE

__all__ = ["make_tools", "FEAT_ENG_SYSTEM_PROMPT_TEMPLATE"]
```

将工具工厂和 prompt 模板作为模块的公开 API 导出。

#### 第 3 层: `registry.py` — Mode 到工具工厂的映射

```python
@dataclass
class ModeSpec:
    tool_factory: Callable[[DataProvider], list[BaseTool]]  # 工具工厂签名
    prompt: str                                             # 系统提示词模板

REGISTRY: dict[Mode, ModeSpec] = {
    Mode.EDA: ModeSpec(
        tool_factory=eda.make_tools,
        prompt=eda.EDA_SYSTEM_PROMPT_TEMPLATE,
    ),
    Mode.FEAT_ENG: ModeSpec(
        tool_factory=feature_engineering.make_tools,
        prompt=feature_engineering.FEAT_ENG_SYSTEM_PROMPT_TEMPLATE,
    ),
}
```

| 关注点                 | 说明                                                              |
|------------------------|-------------------------------------------------------------------|
| `tool_factory` 是**未绑定**的 callable | 存入的是函数对象，尚未传入 `data`，延迟实例化      |
| `REGISTRY` 是全局映射   | 新增 mode 时在此添加一行 `Mode.XXX: ModeSpec(...)` 即可           |

---

#### 第 4 层: `assembly.py` — 图的组装（单一编排点）

`build_graph(data)` 是整个注册链的最终消费端，负责将工具工厂实例化为真实工具并绑入图：

```python
def build_graph(
    data: DataProvider,
    checkpointer: Checkpointer | None = None,
    graph_config: GraphConfig = GraphConfig(),
) -> CompiledStateGraph:
    # ① 遍历 REGISTRY，用传入的 data 实例化所有模式的工具
    tools_by_mode: dict[Mode, list[BaseTool]] = {
        mode: spec.tool_factory(data)   # 此时才调用 make_tools(data)
        for mode, spec in REGISTRY.items()
    }

    # ② 准备 prompt 模板
    prompt_templates: dict[Mode, str] = {
        mode: spec.prompt
        for mode, spec in REGISTRY.items()
    }

    # ③ 公共工具（跨模式可用）
    common_tools = make_common_tools(data)

    # ④ 平铺所有工具给 ToolNode 执行
    all_tools = [*common_tools, *(t for tools in tools_by_mode.values() for t in tools)]

    # ⑤ react_node 按 mode 动态绑定工具子集
    builder.add_node(Node.REACT, partial(
        react_node,
        tools_by_mode=tools_by_mode,
        prompt_templates=prompt_templates,
        common_tools=common_tools,
    ))

    # ⑥ ToolNode 负责执行 LLM 请求的任何工具
    builder.add_node(Node.TOOLS, ToolNode(all_tools, handle_tool_errors=True))
```

---

#### 运行时 per-turn 绑定: `react_node`

`react_node` 每个推理轮次根据 `state["current_mode"]` 选择工具子集：

```python
def react_node(state, *, llm, tools_by_mode, common_tools, ...):
    mode = state["current_mode"]
    tools = [*(common_tools or []), *tools_by_mode[mode]]
    bound = llm.bind_tools(tools)       # 仅当前 mode 的工具对 LLM 可见
    bound.invoke([system_prompt, *messages])
```

#### 新增工具时的注册 checklist

| 步骤 | 文件 | 操作 |
|------|------|------|
| 1 | `tools.py` | 添加 `@tool` 函数，加入 `make_tools()` 返回列表 |
| 2 | `registry.py` | 如果为新 mode，添加 `Mode.XXX: ModeSpec(...)` 映射 |
| 3 | `__init__.py` | 确保 `make_tools` 和 prompt 被导出 |

其余文件（`assembly.py`、`nodes.py`、`wrapper.py`、`cli.py`）均**无需修改**——它们通过遍历 `REGISTRY` 自动发现所有模式及其工具。
